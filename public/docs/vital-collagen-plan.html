Section 01
Executive Summary
Where we are, what we have, and why we can't jump straight to synthesis.
⚡
Bottom Line Up Front
We have 31 documented website changes — not 31 A/B tests. The current backlog is predominantly baseline fixes and small UX corrections. The site has real structural gaps that these items do not address. Before running any synthesis, we need to organize the backlog properly, identify what's missing, and build a second wave of stronger improvements.

Current Situation

A prior session produced 31 documented website changes for Vital Collagen, stored in Airtable. Those records were created without a field distinguishing Direct Changes from A/B Tests. The backlog has no structural clarity around work type, page context, or intent.

The 31 items are mostly valid work — typos, weak copy, missing trust signals, layout friction — but they are overwhelmingly fix-oriented. They address surface-level problems rather than defining how each page should perform as a conversion system. The site needs more than cleanup.

Why These 31 Records Are Not Enough

Baseline fixes remove friction. They are necessary but not sufficient. The site's underlying structure — how trust is built, how products are merchandised, how offers are communicated, how reviews are surfaced — has not been designed yet. The backlog does not reflect this gap.

Why We Should Not Run the Synthesis SOP Yet

🚫
Do Not Run Synthesis
The synthesis SOP is designed to aggregate results from concluded A/B tests — experiments with control groups, traffic splits, and measurable outcomes. We have zero concluded tests. Running synthesis now would produce meaningless output. The SOP should only be used after real experiments have run and returned data.

The Right Next Move

Structure Airtable properly → tag the 31 items by type and page → review each page through a consistent framework → identify what's truly missing → build a second backlog of structural improvements → define page systems → only then design real A/B experiments → run synthesis when those experiments conclude.

Section 04
What's Missing
The current backlog is a cleanup list. Here's the structural work that doesn't exist yet.
🔍
The Problem in Plain Terms
Fixing a broken nav label is not the same as designing navigation. Fixing weak review copy is not the same as building a review system. The current 31 items fix visible errors. None of them define how any page is supposed to work as a conversion system. That architecture doesn't exist in the backlog — and the site can't be optimized meaningfully until it does.
Homepage — Missing Systems6
Hero offer architecture — what is the first-screen value proposition?
First-screen trust structure — social proof placement, review count, authority signals
Best-seller / category entry logic — how does the page direct different intents?
Email or SMS capture strategy — where does list-building happen above the fold?
Brand story compression — a 1–2 sentence narrative that makes the product obvious
Merchandising pathing — which products surface first, and why?
Collection — Missing Systems6
Collection education — header copy that orients the shopper to what they're browsing
Product comparison logic — how does a shopper know which product is right for them?
Sort strategy — correct default sort order and why
Merchandising hierarchy — which products are featured, pinned, or promoted?
Quick-buy / quick-compare — reduce click depth to add-to-cart
Collection-specific intent matching — align copy and imagery to why this category exists
PDP — Missing Systems10
Product media sequence rules — what order do images appear in, and why?
Image order standards — lifestyle first, ingredient second, or vice versa?
Product storytelling system — how does the page move from awareness to conviction?
Benefit hierarchy — what is the #1, #2, #3 reason to buy?
Objection handling system — where do we address price, ingredients, skepticism?
Trust stack structure — which trust signals appear, in what order, at what stage?
Bundle / subscription opportunities
Cross-sell placement rules
Review placement rules — where do reviews sit relative to ATC?
Variant education structure — how does a shopper choose flavor or size?
Cart — Missing Systems6
Upsell strategy — what is offered in cart, in what format, at what price?
Bundle prompts — "add one more to save" or "complete the routine"
Free shipping motivation — threshold progress bar
Reassurance stack — trust signals to reduce abandonment
Better urgency logic — scarcity should only fire under real conditions
Recovery path — what happens when a user removes a product?
Global / Sitewide — Missing Systems7
Image system — rules for how product and lifestyle images are used across pages
Badge system — when do bestseller, new, sale, limited badges appear and under what rules?
Trust system — sitewide trust icon standards, placement rules, and language
Review system — how reviews are surfaced, summarized, and used as conversion assets
Copy system — voice and tone rules, headline patterns, CTA language standards
Offer system — how discounts, bundles, and promotions are communicated sitewide
Mobile layout standards — explicit rules for spacing, tap targets, and component priority

Section 06
The Right Operating Model
Three layers of work. Different timelines. Different outputs. All necessary.

Not all website work is the same type. Running everything as a single flat backlog creates confusion about priority, sequencing, and what "done" means.

1
Layer 1 — Baseline Fixes
Remove obvious friction hurting conversion right now. Typos, broken layouts, missing signals, confusing labels, misfiring logic. These are corrections, not experiments. The outcome is a clean, credible, non-embarrassing site. A necessary prerequisite to everything else — you cannot accurately measure the effect of a structural change on top of a broken foundation.

The current 31 items are almost entirely Layer 1 work.
Active Now — Complete These First
2
Layer 2 — Structural Additions & System Definition
Design how each page is supposed to work. Define the trust system, review system, merchandising hierarchy, offer structure, media sequence rules. Add capabilities not present at all — email capture, bundle prompts, free shipping progress, variant education. These are deliberate design decisions that go live as direct changes once the architecture is agreed on.

This is the next wave of Airtable records. It does not yet exist.
Next — After Layer 1 Is Implemented
3
Layer 3 — A/B Tests & Experimentation
Once foundation is clean and structural systems are in place, run true experiments. A proper A/B test has a clear hypothesis, a control, a treatment, defined metrics, sufficient traffic, and a minimum runtime. Results are what feed the synthesis SOP. Without Layers 1 and 2 complete, experiments produce noisy data and synthesis output is unreliable.

This layer does not exist yet. Do not start it until the layers above are solid.
Future — After Layer 2 Work Is Live
💡
How This Maps to Airtable Workstream Field
Layer 1 → Baseline Fix. Layer 2 → Structural Addition + System Definition. Layer 3 → Experiment. The Workstream field in Airtable gives you a one-click filter to see only the work that belongs in each layer.
Section 07
Recommended Next Steps
The practical sequence. Do these in order. Don't skip ahead.
1
Add the four Airtable fields
Manually create Action Type, Change Intent, Page, and Workstream in Airtable. Set up select options exactly as defined in Section 02. Don't create filtered views yet — do the tagging first.
2
Tag all 31 current records
Bulk-set Action Type → Direct Change and Workstream → Baseline Fix for all 31. Then individually review each record and set the correct Page and Change Intent values. Most will be Fix.
3
Create filtered views and group by Page
Build the filtered views from Section 02. Create one grouped view sorted by Page. This becomes your working view for the next step.
4
Review each page using the 5-lens framework
Work through the Page Review Template in Section 05 for all five page types. This surfaces what the current backlog covers, what it's missing, and where the structural gaps are.
5
Create the second backlog — Structural Additions
Add new records for each missing system in Section 04. Tag as Action Type → Direct Change, Workstream → Structural Addition, Change Intent → Addition / UX System / Merchandising as appropriate.
6
Add System Definition records
For each sitewide system (image system, badge system, trust system, etc.) create a record that documents the decision. Tag as Workstream → System Definition. These are design decisions, not just tasks.
7
Prioritize the combined backlog
With Layer 1 and Layer 2 both visible in Airtable, prioritize by impact and sequencing. Layer 1 items go first. Layer 2 items that unblock other work follow.
8
Design real A/B experiments — then run synthesis
Only after Layer 1 is implemented and Layer 2 systems are live: design true experiments with hypotheses, controls, treatments, and metrics. When those experiments conclude and return data, run the synthesis SOP. Not before.
Section 08
Final Strategic Conclusion
The right order of operations — and why the framing matters.
🎯
The Core Argument
This is not "redesign everything" vs. "optimize everything." That's a false choice. The right approach is sequenced: fix obvious friction, design page systems deliberately, upgrade weak architecture, then test. Each layer makes the next layer more meaningful. Skipping to testing before the structure is solid produces noisy experiments and misleading results.

The Correct Order of Operations

1
Fix Obvious Friction
Remove errors and missing signals that are breaking credibility right now
2
Define Page Systems
Make deliberate decisions about how each page performs as a conversion system
3
Upgrade Architecture
Add what's missing — offers, trust, merchandising, upsell — as structured direct changes
4
Test With Intent
Run real experiments against a solid foundation, then synthesize results
